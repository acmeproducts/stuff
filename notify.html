HTML5 Notification Capability Demo — Enhanced

Two-file GitHub Pages version

This version is designed for root-level hosting such as GitHub Pages. It uses an external JavaScript file named enhanced.js and demonstrates what a browser-based HTML app can do when the page is focused, backgrounded in another tab, or when the browser is open but not the foreground app.

Purpose

The goal is to show, in one place, the maximum practical attention and notification mechanisms available to a plain HTML5 app while the browser remains open. This demo is intended to answer a specific product question: how much signal can a web app provide depending on whether the page is actively viewed, backgrounded in another tab, or hidden behind another application.

The implementation is deliberately split into a root-level HTML file and a root-level JavaScript file so it can be hosted cleanly from a repository root. The JavaScript also creates a blob-backed worker for page-alive background coordination without requiring separate worker assets.

Production interpretation

For a chat app, the strongest reliable mechanisms available while the page is alive are: in-page unread indicators, title changes, favicon badging, optional sound, optional vibration on supported devices, and system notifications if permission is granted. Robust mobile background behavior beyond a live page still requires a same-origin service worker architecture.

Capabilities by state

State What works What the user is likely to notice
Page in focus Banner, inline unread marker, title change, favicon badge, audio, vibration, system notification Everything, because the user is already looking at the page
Another tab active Title change, favicon badge, unread count, system notification, worker-coordinated timers while page is alive Browser UI indicators and any system notification the platform delivers
Another app in foreground System notification may still surface if allowed; in-page indicators continue updating but are not visible until return Mainly system notification; otherwise only deferred indicators on return

Limits

  • A live HTML page can only update what remains loaded and allowed by the browser runtime.
  • System notifications require user permission and may still be handled differently by platform, browser version, and OS policies.
  • App badging support is limited and is strongest for installed web apps/PWAs on supporting platforms.
  • A blob worker helps with page-alive coordination, but it is not a replacement for a service worker.
  • Reliable background delivery when the page is not actively running requires a service worker and usually push.

Blob worker role

This demo uses a dedicated worker created from a blob URL. That keeps the hosted footprint small and proves that page-side work can be delegated cleanly without an extra worker file.

The worker is used here for timing, state reporting, and message emission while the page is alive. It does not create true background notification capability outside the lifecycle boundaries of the page.

Repository hosting assumption

This file expects enhanced.js to sit beside it at the repository root. No path prefix is required.

Core controls

Expected outcome

Permission request will prompt only if the browser allows prompting in the current context. Simulate incoming message triggers all enabled page-alive indicators. Reset returns the page to its clean state.

Focus-state tests

Expected outcome

While the page is focused, banner, title, favicon, and sound effects are immediately observable. Vibration only works on supported devices and browsers.

System-level tests

Expected outcome

System notification depends on permission and browser/platform support. App badge may do nothing on unsupported platforms. Worker ping confirms the blob worker is active and communicating.

Suggested manual scenarios

  • Leave this tab focused and trigger a simulated message.
  • Move to another tab, then trigger a delayed simulated message using the worker timer.
  • Put Chrome in the background behind another app and trigger a delayed simulated message.
  • Return to this page and confirm unread indicators, title state, and event log entries.
Expected outcome

Delayed events are useful for switching tabs or foreground apps before the signal fires. When the browser remains alive, the page and worker should still coordinate the event. Whether the user sees a strong alert depends on notification permission and platform policy.
